# cpcre2 — vendored PCRE2

This directory contains a **vendored copy of [PCRE2](https://github.com/PCRE2Project/pcre2)**, pinned to upstream **10.40**, compiled directly by cgo. There is no external dependency: anyone who runs `go get` / `go build` on this module gets PCRE2 built from these sources — no submodule, no system library, no CMake step.

## Why it lives here

cgo only compiles `.c` files that sit **directly in a package directory** (it does not recurse into subdirectories). That constraint drives the layout below.

```
cpcre2/
  cpcre2.go            Go package stub; carries the #cgo CFLAGS that build PCRE2
  pcre2_*.c   (27)     PCRE2 sources compiled as separate translation units
  LICENCE              Upstream PCRE2 / JIT / sljit license (BSD)
  README.md            This file
  include/             Files that are #included, never compiled standalone:
    pcre2.h            Public header (generated by the upstream build)
    config.h           Build configuration (generated; JIT + Unicode enabled)
    pcre2_*.h          Internal headers
    pcre2_jit_match.c  Include-only JIT sources (pulled in by pcre2_jit_compile.c)
    pcre2_jit_misc.c
    pcre2_ucptables.c  Include-only (pulled in by pcre2_tables.c)
    sljit/             Stack-less JIT backend (#included by pcre2_jit_compile.c)
```

- The **27 `pcre2_*.c`** at the top level are the real translation units, matching what the upstream CMake build compiles.
- Everything under **`include/`** is only ever reached through `#include` directives (e.g. `pcre2_jit_compile.c` does `#include "sljit/sljitLir.c"`, `pcre2_tables.c` does `#include "pcre2_ucptables.c"`). Compiling them as standalone files would trip their `#error "... must be included from ..."` guards, so they are kept out of the package directory on purpose.

## How it is built

`cpcre2.go` declares the compile flags; cgo handles the rest during a normal `go build`:

```go
#cgo CFLAGS: -I${SRCDIR}/include -DHAVE_CONFIG_H -DPCRE2_CODE_UNIT_WIDTH=8 -DPCRE2_STATIC
```

- `-Iinclude` — find `pcre2.h`, `config.h`, and the include-only sources
- `HAVE_CONFIG_H` — use the bundled `include/config.h` (has `SUPPORT_JIT=1`, `SUPPORT_UNICODE=1`)
- `PCRE2_CODE_UNIT_WIDTH=8` — UTF-8 / 8-bit build
- `PCRE2_STATIC` — no DLL import/export decorations

The package exports **no Go API**. The parent `pcre2` package blank-imports it (`_ "github.com/CoozillaX/g79-regex/internal/pcre2/cpcre2"`) so the object code is linked in, then calls the PCRE2 C symbols from its own `bridge.c`.

The `.clangd` files in `include/` and `include/sljit/` suppress IDE diagnostics for the include-only `.c` files — those are expected to "fail" when opened as standalone units and do not affect the actual build.

## Updating PCRE2

The pinned generated files (`include/pcre2.h`, `include/config.h`, `pcre2_chartables.c`) come from an upstream build tree. To move to a new release:

1. Check out the desired tag from upstream PCRE2 and configure it once (CMake or autotools) to generate `pcre2.h`, `config.h`, and `pcre2_chartables.c`.
2. Copy the 27 `pcre2_*.c` (compiled set) into this directory, and the headers / include-only `.c` / `sljit/` into `include/`.
3. Refresh `LICENCE` and update the version noted in `cpcre2.go` and at the top of this file.
4. Run `go build ./...` and `go test ./review/` to verify (JIT + Unicode should still work).
